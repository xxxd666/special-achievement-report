# 亚马逊 Amazon · 6-Pager 叙事备忘 + PR/FAQ（Working Backwards）

> 亚马逊开会不用 PPT，用**6 页叙事备忘**（完整句子写成的文档，开会先静默读 20 分钟）；做新东西先用**Working Backwards**——倒着从"成功后对外的新闻稿 + 常见问答"写起。
> 在本技能里：当用户做**立项 / 深度方案论证 / 需要严密思考**，或汇报对象习惯读文档而非看 PPT 时用它。**这套天然更契合形态①（深度阅读 HTML），不一定要做横向 PPT。**

## 一图看懂

```
PR/FAQ（Working Backwards）          6-Pager（叙事备忘）
先写"成功后的新闻稿"                 用完整段落讲清楚：
  ↓ 倒推                            背景→问题→方案→数据→权衡→下一步
再写客户/领导会问的 FAQ              （不是 bullet，是能读的文章）
  ↓                                 开会先静默读，再讨论
确认这事值得做，再动手
```

## 核心原则

| 原则 | 说明 | 常见错误 |
|------|------|---------|
| **叙事而非要点** | 用完整句子和段落，逼自己想清楚 | 用 bullet 掩盖逻辑漏洞 |
| **Working Backwards** | 从结果/用户视角倒推该做什么 | 从"我们有什么"正推，做偏 |
| **PR/FAQ** | 先写新闻稿 + 预想问答，验证价值 | 直接进方案，没想清楚为谁解决什么 |
| **数据嵌在叙事里** | 论点旁边就给数据支撑 | 叙事和数据两张皮 |

## 标准产物结构

**6-Pager（约 6 页叙事）：**
```
1. 背景与问题（为什么现在要做这事）
2. 我们的方案（一句话讲清，再展开）
3. 关键数据/证据
4. 权衡与取舍（讨论过的其他选项、为什么不选）
5. 风险与应对
6. 下一步与所需支持
```

**PR/FAQ（立项用）：**
```
新闻稿：假装事情已成功，写一段对外公告（标题+导语+引述）
FAQ：列领导/用户最可能问的 8-10 个问题 + 回答
```

## 行政/园区运营实操例子

给"智慧园区 2.0 上线"写一段 PR（先想成功长什么样）：

> **【园区动态】智慧园区 2.0 上线，报修响应进入"分钟级"**
> 即日起，园区全面启用智慧工单系统。员工扫码即可报修，系统自动派单到最近的工程人员，平均响应从 38 分钟压缩到 12 分钟。运营负责人表示："过去靠电话和人盯，现在靠系统和数据，员工体验和我们的效率都上了一个台阶。"

配套 FAQ：
- 问：和老流程比省了多少人力？答：……
- 问：投入多少、多久回本？答：……
- 问：员工不会用怎么办？答：……

## 适用 vs 什么时候别用

| 适合 | 别用 |
|------|------|
| 立项、深度方案、要严密论证 | 现场要快速翻页演讲（叙事 memo 不适合投屏） |
| 对象习惯读文档、要留档细看 | 时间极短、只要一句结论（用 BLUF） |
| 想逼自己把逻辑想透 | 纯数据看板类汇报 |

> 配合：6-Pager 用形态①（纵向 HTML）承载最合适；PR/FAQ 可作为立项汇报的开篇钩子，再接麦肯锡骨架展开论证。
