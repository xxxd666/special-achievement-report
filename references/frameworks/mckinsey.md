# 麦肯锡方法论（默认基线 · 9 种 / 3 子类，组合使用）

> 这是本技能的**默认分析框架**。麦肯锡不是单一一招，而是一套可叠加的方法。
> 真实汇报里，你总是**几种方法一起用**：先定怎么讲（叙事），再搭骨架（结构），最后把数据变成洞见和行动（分析）。

## 怎么用这一套（先看这里）

把 9 种方法按用途分成 3 个子类。一次汇报通常每个子类各取 1–2 个：

| 子类 | 含哪几种 | 解决什么 |
|------|----------|----------|
| **A · 叙事 / 定位** | ① SCQA 叙事　② 序言结构（4 变体） | 怎么开场、用什么节奏把人带进来 |
| **B · 结构化拆解** | ③ 金字塔原理　④ MECE　⑤ 议题树　⑥ 逻辑树 | 怎么搭骨架、把事情分清楚不重不漏 |
| **C · 分析 / 洞察** | ⑦ 空·雨·伞　⑧ 假设驱动　⑨ 80/20（帕累托） | 怎么从一堆数据里逼出结论和下一步 |

**默认组合 = SCQA + 金字塔 + MECE + 空·雨·伞**
覆盖「开场 → 骨架 → 分维 → 出行动」这条最稳的主线，适配绝大多数向上汇报。
需要时再加减：时间紧就把 SCQA 换成开门见山式；事项太多就加 80/20 抓重点；要诊断根因就加议题树。

# A · 叙事 / 定位

## ① SCQA 叙事 + ② 序言结构（4 变体）

SCQA 是开场的四块积木：
- **S 背景**：大家都认同的现状（"公司今年从 3000 人涨到 5000 人"）
- **C 冲突**：打破现状的变化或矛盾（"工位缺口 40%，会议室日均冲突 65%"）
- **Q 问题**：由冲突自然引出的问题（"怎么在不加预算下保住办公体验？"）
- **A 答案**：你的解法，也就是整份汇报的主线（"空间改造 + 智能调度 + 远程协同"）

**4 种变体——按"给谁看、有多少时间"选：**

| 变体 | 结构 | 优势 | 风险 | 什么时候用 |
|------|------|------|------|-----------|
| **标准式** | S→C→Q→A | 叙事完整、逻辑严密 | 前摇偏长 | 书面报告、要存档细读 |
| **开门见山式** | S→C→A | 快速进重点 | 少了问题引导 | 领导时间紧、年终汇报 |
| **突出忧患式** | C→Q→A | 冲击力强、引重视 | 可能显得负面 | 争取资源 / 推动改变 |
| **突出信心式** | Q→A | 直接给答案 | 缺背景铺垫 | 已有共识基础 |

# B · 结构化拆解

## ③ 金字塔原理 — 结论先行

### 4 个核心原则

| 原则 | 说明 | 常见错误 |
|------|------|---------|
| **结论先行** | 每层先说结论，再展开论据 | 先罗列过程，最后才给结论（领导没耐心看完） |
| **以上统下** | 上层思想是下层思想的概括 | 上层标题是"A"，下层内容却是 B+C |
| **归类分组** | 每组思想属于同一逻辑范畴 | 把"降本"和"员工活动"放一组 |
| **逻辑递进** | 按演绎或归纳顺序排列 | 跳跃式排列，看不出逻辑关系 |

### 纵向关系：问答式对话

```
┌──────────────────────────────────┐
│ 结论：我们应关闭 3 个低效会议室  │  ← "所以呢？"
└──────────────┬───────────────────┘
               │ "为什么？"
    ┌──────────┼──────────┐
┌───┴───┐ ┌───┴───┐ ┌───┴───┐
│利用率  │ │维护成 │ │员工反 │
│<20%   │ │本占比高│ │馈差   │  ← "为什么？"
└───┬───┘ └───┬───┘ └───┬───┘
    │ "怎么知道?"   │        │
 数据+调研    财务数据   问卷数据  ← 事实支撑
```

### 横向关系：归纳 vs 演绎

- **归纳推理**（推荐用于汇报）：从多个具体事实中提炼共性结论
  - "项目 A 满意度提升 22%，项目 B 提升 18%，项目 C 提升 15% → 我们的服务质量整体在提升"
- **演绎推理**（用于需要严密论证的场景）：大前提 → 小前提 → 结论

## ④ MECE 切分实战矩阵 — 不重不漏

### 对职能部门工作的 6 种经典切法

#### 切法 1：按业务模块（最常用）

```
职能部门工作全景
├── 空间管理（办公空间/会议室/仓库/车位）
├── 设施运维（水电/空调/网络/消防/电梯）
├── 行政服务（前台/收发/文印/用车/差旅）
├── 安全管控（门禁/监控/消防/防疫/应急）
├── 员工体验（活动/福利/食堂/健身房/茶水间）
└── 采购与供应商管理（招标/合同/考核/付款）
```

#### 切法 2：按价值产出（让领导看到 ROI）

```
年度成果
├── 降本（直接成本节约）
│   ├── 供应商重新招标 → 节省 280 万
│   ├── 能耗优化 → 节省 95 万
│   └── 空间利用率提升 → 避免新增租赁 500 万
├── 提效（效率提升）
│   ├── 智慧工单系统 → 响应时间 ↓60%
│   ├── 线上审批 → 流程从 3 天 → 2 小时
│   └── 自助服务终端 → 减少人工咨询 70%
├── 体验（满意度提升）
│   ├── 餐厅改造 → 满意度 3.2→4.5
│   ├── 健身房升级 → 使用率 ↑300%
│   └── 办公环境优化 → 离职意向 ↓15%
└── 风险（合规安全）
    ├── 消防改造 → 通过消防验收
    ├── 疫情防控 → 0 聚集性感染
    └── 供应商合规 → 审计 0 问题项
```

#### 切法 3：按时间线（适合过程复盘）

```
项目全生命周期
├── 准备期（Q4 2024）
│   ├── 需求调研：访谈 12 位部门负责人
│   ├── 方案设计：3 轮方案迭代
│   └── 预算申请：获批 580 万
├── 建设期（Q1-Q2 2025）
│   ├── 施工改造：面积 3,200㎡，历时 45 天
│   ├── 系统部署：智慧园区平台上线
│   └── 供应商切换：完成 8 家核心供应商更替
├── 运营期（Q3-Q4 2025）
│   ├── 平稳运行：SLA 达标率 99.2%
│   ├── 持续优化：15 项微改进落地
│   └── 满意度回访：4.2/5，目标 3.8 超额完成
└── 复盘期（Q1 2026）
    ├── 经验沉淀：形成 3 份 SOP
    └── 推广应用：方案复制到 2 个分公司
```

#### 切法 4：按服务对象

```
服务矩阵
├── 面向员工（入离职办理 / 日常办公服务 / 活动福利）
├── 面向管理层（预算执行 / 空间规划 / 成本分析）
├── 面向外部（访客接待 / 供应商管理 / 政府物业关系）
└── 面向业务部门（特殊需求响应 / 跨部门协作项目）
```

#### 切法 5：按资源投入

```
成本结构 MECE 分析
├── 人力成本（编制内 + 外包）
├── 空间成本（租金 + 物业 + 装修摊销）
├── 运营成本（水电能耗 / 设施维护 / 行政物料 / 网络通讯）
├── 项目成本（活动执行 / 系统采购开发 / 咨询服务）
└── 其他（保险 / 税费 / 备用金）
```

#### 切法 6：SWOT 式（用于未来规划）

```
当前状态
├── 优势（可复制的成功）
├── 劣势（需要改进的不足）
├── 机会（外部趋势可利用的）
└── 威胁（外部风险需防范的）
```

## ⑤ 议题树 — 问题结构化拆解

### 3 种形态

#### 类型 1：为什么树（Why Tree）— 诊断问题

```
问题：员工满意度为什么下降？
├── 服务层面
│   ├── 响应变慢？（响应时间 15min→22min ✓）
│   ├── 质量下降？（一次解决率 91%→89% 略降）
│   └── 态度变差？（投诉中态度相关 15%→35% ✗ 这是主因）
├── 环境层面
│   ├── 工位拥挤？（面积/人 4.5㎡ 不变 ✗）
│   ├── 设施老化？（空调报修 ↑120% ✓ 这是主因）
│   └── 配套不足？（餐厅排队 8min→18min ✓）
└── 沟通层面
    ├── 渠道不畅？（知道报修渠道的仅 65% ✓）
    └── 反馈不闭环？（工单回访率 22% ✗）
```

#### 类型 2：怎么办树（How Tree）— 寻找方案

```
问题：如何在 3 个月内提升园区服务满意度 10%？
├── 短期速效（当月见效）
│   ├── 响应提速：引入抢单模式 + 30min SLA
│   ├── 痛点攻坚：空调集中检修 + 备用机部署
│   └── 沟通强化：周报 + 告示 + 全员群更新机制
├── 中期建设（1-3 月）
│   ├── 系统上线：智慧工单系统 + 移动端报修
│   ├── 流程再造：建立"首问负责制"
│   └── 供应商考核：引入 SLA 考核 + 奖惩机制
└── 长期优化（3-6 月）
    ├── 空间升级：3F 办公区改造
    ├── 服务产品化：制定行政服务目录
    └── 团队赋能：服务意识 + 专业技能培训
```

## ⑥ 逻辑树 — 结构/议题/假设三类树的统称

逻辑树是上面议题树的更一般形态，按目的分三种，选对种类比画得漂亮更重要：

| 逻辑树类型 | 用途 | 一句话 |
|-----------|------|--------|
| **结构树** | 把一个整体拆成不重不漏的部分 | "成本由哪些部分构成？"（≈ MECE 的可视化） |
| **议题树（Why/How）** | 诊断原因 / 找解决方案 | 见上「议题树」 |
| **假设树** | 先立假设，再去验证 | 见下「假设驱动」的假设验证树 |

> 实操建议：汇报里画树时只展开到"能落到数据"的那一层，再深就是给自己看的草稿，不必上汇报。

# C · 分析 / 洞察

## ⑦ 空·雨·伞 — 从数据到行动

### 三层递进（汇报价值的分水岭）

```
┌─────────────────────────────────────────────────┐
│  🟢 空（初级）："今年做完了 50 件事"             │
│     ↑ 60% 的人停在这里                            │
│  🟡 雨（中级）："这 50 件里，3 件最关键，         │
│     因为它们解决了 XX 瓶颈"                       │
│     ↑ 30% 的人做到                                │
│  🔴 伞（高级）："据此建议明年重点投 A 方向，      │
│     预计降本 300 万；B 方向建议暂停，ROI 不达预期" │
│     ↑ 只有 10% 的人能做到                         │
└─────────────────────────────────────────────────┘
```

### 职能部门常用空雨伞对照表

| 空（事实） | 雨（解读） | 伞（行动） |
|-----------|-----------|-----------|
| 2025 年能耗费用同比 ↑8% | 员工增长 40% + 新增 24h 数据机房是主因 | 2026 推进机房独立计量 + 余热回收，目标回落至 2024 水平 |
| 工单满意度 4.2 分 | 超行业基准 3.8，但离"卓越"4.5 仍有差距 | Q1 主攻"等待时间"，引入抢单 + 超时预警 |
| 供应商履约准时率 87% | 2 家核心供应商拖后腿，拉低整体 | Q1 启动核心供应商季度述职 + 末位淘汰 |
| 员工活动参与率 35%→28% | 大型活动下降，小而美兴趣小组反升 | 2026 活动策略从"大而全"转向"精而美"社群运营 |
| 跨部门"响应速度"仅 3.1 分 | 行政需求没有 SLA 承诺，沟通成本高 | 建行政服务目录 + SLA，关键需求 2h 内响应 |

### 怎么练出"伞"的思维

1. **向上追问**：每个数据后面问一句"所以呢？该怎么办？"
2. **领导视角**：如果我是领导，看完这页最想问什么？
3. **ROI 思维**：这个行动，投入 vs 产出是什么？

## ⑧ 假设驱动 — 先立假设，再验证

不要等数据全收齐才下结论。先用经验大胆立一个假设，再有针对性地找证据，效率高得多。

### 假设验证树（Hypothesis Tree）

```
假设：设施老化是报修量激增的主因
├── 验证1：设备年限 → 报修设备平均 8.2 年 vs 行业更换周期 6-8 年 ✓
├── 验证2：报修分布 → 空调占报修 45%，其中用 10 年以上占 72% ✓
├── 验证3：维护记录 → 2024 仅完成计划维护 60%，2025 降至 42% ✓
└── 验证4：横向对比 → B 栋（2019 翻新）报修量仅为 A 栋（2014）的 35% ✓
    ✅ 假设成立：设施老化 + 维护不足 = 主因
```

> 用在汇报里：把"假设 + 几条验证 + 结论"压成一页，比铺陈全部数据更有说服力。

## ⑨ 80/20（帕累托）— 抓住关键少数

事项一多就容易"什么都讲、什么都没讲透"。用 80/20 先排序：**哪 20% 的事贡献了 80% 的价值？** 汇报就重点展开那 20%，其余一笔带过或附录。

- 例：全年 101 项改造，其中 12 项贡献了 75% 的降本 → 汇报主体只讲这 12 项，剩下 89 项用一句"其余 89 项常规维保，累计降本 XX 万"带过。
- 配合金字塔：被选出的关键少数，正好充当金字塔第二层的几个"成果维度"。

# 汇报叙事路径选择指南（把 A/B/C 串起来）

```
你的汇报目的是什么？
│
├── 展示成果，争取认可 ──→ SCA 开门见山式
│   适用：年终总结、季度汇报
│   结构：背景→挑战→成果（重点展开）→未来规划
│
├── 复盘项目，总结经验 ──→ SCQA 标准叙事式
│   适用：项目复盘、专项汇报
│   结构：背景→目标→过程拆解→成败分析→经验沉淀
│
├── 暴露问题，争取资源 ──→ CQA 突出忧患式
│   适用：资源申请、风险预警
│   结构：危机描述→不做会怎样→方案对比→资源需求
│
└── 汇报进展，同步信息 ──→ 金字塔直给式
    适用：周报/月报、状态同步
    结构：结论→数据看板→风险项→下一步
```

## 标准产物结构（默认大纲）

用默认组合（SCQA + 金字塔 + MECE + 空雨伞）产出的汇报，通常是这 5 页：

```
P1 · Executive Summary   一句话核心结论 + 3 个核心数字（金字塔塔尖 + SCQA 的 A）
P2 · 结果全景            目标 vs 实际 + 关键里程碑（金字塔第二层概览）
P3 · 过程拆解           3 大关键举措，MECE 分维 + 80/20 抓重点
P4 · 洞察与经验沉淀      空·雨·伞：事实 → 解读 → 行动
P5 · 未来展望           下一步主攻方向 + 资源需求（伞的延伸）
```

## 什么时候别只用麦肯锡

- 只要一句结论给大忙人 → 加 **BLUF**（见 beyond-pyramid.md）
- 纯目标达成复盘 → 用 **Google OKR**
- 审计 / 自查 / 挑问题 → 用 **PwC findings**
- 转型 / 流程改造 → 用 **Deloitte As-Is/To-Be**
（这些都可与麦肯锡组合，见 framework-selector.md）
