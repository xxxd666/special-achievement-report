# Bain · Answer First（答案先行）+ 结果交付（Results Delivered）

> 贝恩最讲"为客户创造可衡量的结果"。汇报上体现为两点：**开口就给答案**（不让人等）、**一切围绕"带来了什么结果"**（不堆过程）。
> 在本技能里：当用户要"争取认可/资源、突出 ROI、让领导第一眼看到价值"时用它。

## 一图看懂

```
传统（倒着讲）              Bain（正着给）
背景 → 过程 → 分析 → 结论    结论/结果 ↑ 先抛出来
（领导翻到最后才看到价值）    └ 然后才是支撑它的过程与分析
                            "我们做到了 X，因为做了 A/B/C"
```

## 核心原则

| 原则 | 说明 | 常见错误 |
|------|------|---------|
| **Answer First** | 第一屏就给结论 + 核心结果数字 | 先讲背景和苦劳，结果埋在最后 |
| **结果导向** | 每件事都回答"带来了什么可衡量结果" | 罗列"做了什么"，不说"产生了什么" |
| **电梯测试** | 30 秒能讲完核心：做了什么、值多少 | 讲了 5 分钟还没说到价值 |
| **量化交付** | 结果尽量带数字、带对比（vs 目标/去年） | "效果不错""有所提升"这类虚词 |

## 标准产物结构

```
P1 答案页：一句话结论 + 3 个最硬的结果数字（降本 X / 提效 Y / 满意度 Z）
P2 结果全景：目标 vs 实际，全部兑现情况一览
P3-Pn 支撑：每块讲"做了什么动作 → 交付了什么结果"，动作服务于结果
P尾 下一步：要继续/扩大，需要什么资源（顺势要资源）
```

## 行政/园区运营实操例子

开篇答案页（而不是从"今年园区工作总结"讲起）：

> **"全年园区运营在人员增长 40% 的压力下，实现人均办公成本 ↓15%、服务满意度 ↑22%、安全事故 0 起——靠的是供应商重招、智慧工单、空间改造三件事。"**
> （三个结果数字先砸出来，后面每页再展开是怎么做到的。）

每个动作都翻译成结果：
- 重招供应商 → **省 280 万**（不是"完成了 8 家供应商更替"）
- 上智慧工单 → **响应提速 60%**（不是"系统已上线"）

## 适用 vs 什么时候别用

| 适合 | 别用 |
|------|------|
| 争取认可/预算/资源，对象是高管 | 结论还没定、需要和领导一起讨论推演 |
| 结果硬、数字漂亮，想第一时间亮出来 | 结果还很弱、主要想讲过程艰辛（答案先行会暴露结果不足） |
| 时间短、要电梯式讲完 | 需要完整还原思考过程的复盘 |

> 配合：Bain 定"先给答案"的姿态，BCG 定每页标题，麦肯锡补骨架。三者常一起用在高管汇报。
