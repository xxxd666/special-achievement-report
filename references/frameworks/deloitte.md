# 德勤 Deloitte · As-Is → To-Be + 价值地图

> 德勤做转型/流程咨询的核心套路：先把**现状（As-Is）**画清楚，再描**目标态（To-Be）**，中间是**差距**，然后用**价值地图**把每个改造举措连到它创造的业务价值上。
> 在本技能里：当用户做**转型规划 / 流程改造 / 数字化 / 申请资源建设**时用它——它天然适合"我们要从现在的样子，变成更好的样子"这类叙事（**你那张园区工作流改造图，正是这个场景**）。

## 一图看懂

```
As-Is（现状）        →    差距/痛点    →    To-Be（目标态）
人工逐项、串行、靠经验      慢、易错、难复制      AI 测算/制单、并行、数据驱动
        │                                          │
        └──────────── 价值地图：每个举措 → 连到 → 降本/提效/体验/风险 ─┘
```

## 核心原则

| 原则 | 说明 | 常见错误 |
|------|------|---------|
| **As-Is 要诚实** | 现状如实画，痛点点出来 | 美化现状，显不出改造必要性 |
| **To-Be 要具体** | 目标态可感知、可衡量 | 目标空泛（"更高效""更智能"） |
| **差距即理由** | 现状与目标的差距 = 为什么要投入 | 跳过差距，直接要资源 |
| **价值地图闭环** | 每个举措都连到一个业务价值 | 举措一堆，说不清值在哪 |

## 标准产物结构

```
P1 As-Is：现状全景 + 关键痛点（流程图/现状盘点）
P2 To-Be：目标蓝图 + 可衡量的目标态
P3 差距分析：从 As-Is 到 To-Be 差什么（能力/系统/人/钱）
P4 价值地图：每个改造举措 → 对应的降本/提效/体验/风险价值
P5 路线图：分期落地 + 资源需求 + 里程碑
```

## 行政/园区运营实操例子

园区工程改造工作流的 As-Is / To-Be：

| 维度 | As-Is（现状） | To-Be（目标态） |
|------|------|------|
| 造价测算 | 人工逐项比对，30 min/单 | AI 初筛偏差，5 min/单，人工只复核 |
| 制单 | 手工填《工程联系单》，20–30 min | 读 Excel 自动生成，3 min |
| 成果汇报 | 人工找资料排版，2–3 天 | 方法论 + 模板，2–3 小时 |
| 整体 | 串行、靠经验、难复制 | 并行、数据驱动、可沉淀复用 |

价值地图（举措 → 价值）：
- AI 测算初筛 → **提效 83% + 降本**（减少测算误差）
- AI 制单 → **提效 88% + 标准化**（口径统一）
- 方法论汇报 → **决策质量 + 经验沉淀**

## 适用 vs 什么时候别用

| 适合 | 别用 |
|------|------|
| 转型/改造/数字化，要讲"从旧到新" | 单纯回顾过去成果（它重在未来设计） |
| 申请资源建设，需要论证差距和价值 | 极短汇报（As-Is/To-Be 需要篇幅铺开） |
| 流程优化、能力建设类项目 | 目标态还没想清楚时（To-Be 会很虚） |

> 配合：As-Is/To-Be 适合做麦肯锡汇报的"现状→未来"主线；价值地图可与空雨伞的"伞（行动）"对接；视觉上 As-Is/To-Be 用左右对比版式（瑞士风 S08 Duo Compare）最直观。
